**Status:** done (2026-06-13)

## Why

Every issue must belong to a project. Users need a dedicated project workspace so global administration navigation does not consume space while working inside a project. Project stewards must manage who can access each project workspace.

## Behavior delta

**Before**

- Global sidebar included direct access to a single **Issues list** for all issues.
- Post-sign-in default screen was **Issues list**.
- No project entity, permissions, or workspace layout.

**After**

- Sidebar entry **Projects** opens **Projects list**; issues are listed only as **Project issues list** inside a project workspace.
- Post-sign-in default (after compliance gates) is **Projects list**.
- Permissions **Projects.View** and **Projects.Manage** control project screens; default **User** role receives **Projects.View**.
- Project workspace replaces global sidebar with Overview / Issues / Settings navigation and a single application header.
- **Project settings** supports **Add member**, inline **Role** changes, and **Remove** for users with **Projects.Manage**.

## Implementation scope (optional)

- Backend: Project aggregate, Issue.ProjectId, permissions, CRUD and member APIs, migration with default **General** project.
- Frontend: Projects feature, app-shell project workspace mode, project-scoped issue routes, **Project settings** member management UI.
- Tests: backend unit/integration (including project member endpoints), frontend unit for **Project settings**, E2E smoke updates for project-scoped flows.
