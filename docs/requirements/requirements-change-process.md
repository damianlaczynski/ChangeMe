# Requirements change process

> Workflow for analysts and developers: pending deltas, validation, and handoff. Trimmed after delivery.

## Start here

| Role                                   | Read first                                                         | Then                                                                                                                                       |
| -------------------------------------- | ------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------ |
| **Analyst** (new or updated `FR-*`)    | [requirements-authoring-guide.md](requirements-authoring-guide.md) | Edit specs → [\_changes-template.md](_changes-template.md) → `npm run requirements:validate`                                               |
| **Developer** (implement pending work) | Pending files in [changes/](changes/)                              | Linked `FR-*` + `_shared/` → [feature-recipes.md](../guides/feature-recipes.md) + [testing-guidelines.md](../guides/testing-guidelines.md) |
| **Find a specification**               | [README.md](README.md) (auto-generated index)                      | Open the linked `FR-*` file                                                                                                                |

## Document types

| Location                                                          | Role                                                                                               |
| ----------------------------------------------------------------- | -------------------------------------------------------------------------------------------------- |
| `docs/requirements/functional/<domain>/fr-<area>-<nnn>-<slug>.md` | **Functional specification** (`FR-*`) — atomic, testable behavior per screen or flow.              |
| `docs/requirements/_shared/reference/*.md`                        | **Reference** — glossary, account model, permissions.                                              |
| `docs/requirements/_shared/functional/*.md`                       | **Shared functional patterns** (`FR-UI-001`, UI/UX defaults).                                      |
| `docs/requirements/_shared/non-functional/*.md`                   | **Non-functional requirements** (`NFR-*`) — accessibility, i18n, performance, responsiveness.      |
| `docs/requirements/changes/YYYY-MM-DD-<title>.md`                 | **Pending delta** — one file per coherent batch until implemented.                                 |
| `docs/requirements/requirements-authoring-guide.md`               | **Authoring** — how to write and update `FR-*` specifications (this document’s analyst companion). |
| `docs/requirements/_functional-specification-template.md`         | **Template** — skeleton to copy when adding a new `FR-*` file.                                     |
| `docs/requirements/_changes-template.md`                          | **Template** — copy when adding a pending change record.                                           |
| `docs/requirements/README.md`                                     | **Index** — manifest (regenerated by `npm run requirements:validate`).                             |
| `docs/requirements/requirements-change-process.md`                | **Process** — workflow, roles, change records, validation (this document).                         |

Long-term history lives in git (specification edits and past change records in commit history).

Domain folders under `functional/` and their specifications are listed in [README.md](README.md) (regenerated by validation). When adding a new domain folder, run `npm run requirements:validate` to refresh the index.

## Roles

### Analyst (or author of the change)

1. Read `docs/requirements/requirements-authoring-guide.md` when creating or substantially updating `FR-*` files.
2. Decide whether the change touches **shared concepts** (terms, gates, permissions, NFR, UI patterns). If yes, edit the matching file under `docs/requirements/_shared/` **once** instead of duplicating across functional specifications.
3. Edit only the affected `FR-*` files under `docs/requirements/functional/<domain>/`. For **new** specifications, copy `docs/requirements/_functional-specification-template.md` to the target path and follow the authoring guide.
4. Create **one** file in `docs/requirements/changes/` from `docs/requirements/_changes-template.md` (filename: `YYYY-MM-DD-short-title.md`).
5. List every touched `FR-*`, `NFR-*`, and shared doc in the change record; describe the **behavior delta** (not a copy of the specification).
6. Run `npm run requirements:validate` before opening a PR.

### Developer

1. Read pending files in `docs/requirements/changes/` (**Status:** `pending`).
2. Read the linked `FR-*` files and any `_shared/` docs referenced there.
3. Implement and verify against **Functional requirements** and inherited patterns (use `docs/guides/feature-recipes.md` for technical steps; `docs/guides/testing-guidelines.md` for automated checks).
4. When the work is merged, set **Status:** `done (YYYY-MM-DD)` on the change file or **delete** it.

An empty changes queue (only `README.md`) is normal.

## Functional specification structure

Section layout, writing rules, and new-vs-updated guidance: [requirements-authoring-guide.md](requirements-authoring-guide.md). Skeleton: [\_functional-specification-template.md](_functional-specification-template.md).

## Sections in a change record

| Section                               | Purpose                                                |
| ------------------------------------- | ------------------------------------------------------ |
| **Why**                               | Brief intent. Optional.                                |
| **Functional specifications touched** | Which `FR-*` files are new or updated; one line each.  |
| **Non-functional documents touched**  | Which `NFR-*` or `_shared/` files changed, if any.     |
| **Behavior delta**                    | What is different from the old product (Before/After). |
| **Implementation scope**              | Optional hint for backend/frontend/test areas.         |

Do not add: deploy notes, source code paths, API endpoint lists, or implementation checklists (those belong in design docs or PR description).

## When to add a change record

One record per coherent batch of specification changes (feature slice). Skip typo-only edits that do not change behavior.

## Cross-domain changes

Use **one change file** that lists all touched `FR-*` across domains. Do not split into per-domain changelog files.

## Validation

From the repository root:

```bash
npm run requirements:validate
```

Checks: `FR-*` / `NFR-*` frontmatter, required FR sections, broken cross-references, shared docs present, change records reference valid IDs. Regenerates `docs/requirements/README.md` and `.requirements-manifest.json`.

## For AI agents

1. When authoring or editing `FR-*` files, follow `docs/requirements/requirements-authoring-guide.md`.
2. If `docs/requirements/changes/` has files with **Status:** `pending`, read them plus referenced `FR-*` and `_shared/` paths.
3. Otherwise use `docs/requirements/README.md` to locate relevant functional specifications.
4. Cross-cutting terms and gate ordering: read `docs/requirements/_shared/reference/` before inferring behavior from a single `FR-*`.
5. Shared UI and NFR defaults: read `docs/requirements/_shared/functional/ui-patterns.md` (`FR-UI-001`) and `docs/requirements/_shared/non-functional/product-quality.md` (`NFR-QUAL-001`).
