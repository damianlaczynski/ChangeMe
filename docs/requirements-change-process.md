# Requirements change process

> Pending requirement changes for developers; trimmed after delivery.

## Document types

| Location                                       | Role                                                                        |
| ---------------------------------------------- | --------------------------------------------------------------------------- |
| `docs/req/<domain>/req-<area>-<nnn>-<slug>.md` | **Specification** — one atomic, testable REQ per file.                      |
| `docs/req/_shared/*.md`                        | **Cross-cutting** — glossary, account model, compliance gates, permissions. |
| `docs/req/changes/YYYY-MM-DD-<title>.md`       | **Pending delta** — one file per coherent batch until implemented.          |
| `docs/req/README.md`                           | **Index** — manifest of all REQs (regenerated by `npm run req:validate`).   |

Long-term history lives in git (REQ edits and past change records in commit history).

## Domains

| Folder         | Scope                                                   |
| -------------- | ------------------------------------------------------- |
| `identity/`    | Auth, sessions, password, 2FA, OIDC, email verification |
| `users/`       | User admin, my account, deactivation                    |
| `invitations/` | Invite, resend, cancel, status                          |
| `access/`      | Roles and permissions                                   |
| `passkeys/`    | WebAuthn passkeys                                       |
| `projects/`    | Projects and issue project membership                   |
| `issues/`      | Issues, attachments, notifications                      |
| `time/`        | Work time logging, reports, and audit                   |
| `billing/`     | Employment, contracts, leave, settlements, and reports  |

## Roles

### Analyst (or author of the REQ change)

1. Decide whether the change touches **shared concepts** (terms, gates, permissions, account model). If yes, edit `docs/req/_shared/` **once** instead of duplicating in multiple REQs.
2. Edit only the affected atomic REQ files under `docs/req/<domain>/`.
3. Create **one** file in `docs/req/changes/` from `docs/req/changes/_template.md` (filename: `YYYY-MM-DD-short-title.md`).
4. List every touched REQ and shared doc in the change record; describe the **behavior delta** (not a copy of the REQ).
5. Run `npm run req:validate` before opening a PR.

### Developer

1. Read pending files in `docs/req/changes/` (**Status:** `pending`).
2. Read the linked REQ files and any `_shared/` docs referenced there.
3. Implement and test against the REQ (use `docs/feature-recipes.md` for technical steps).
4. When the work is merged, set **Status:** `done (YYYY-MM-DD)` on the change file or **delete** it.

An empty changes queue (only `_template.md` and `README.md`) is normal.

## Sections in a change record

| Section                  | Purpose                                                |
| ------------------------ | ------------------------------------------------------ |
| **Why**                  | Brief intent. Optional.                                |
| **Requirements touched** | Which `REQ-*` files are new or updated; one line each. |
| **Shared docs touched**  | Which `_shared/` files changed, if any.                |
| **Behavior delta**       | What is different from the old product (Before/After). |
| **Implementation scope** | Optional hint for backend/frontend/test areas.         |

Do not add: deploy notes, file paths in source code, API endpoint lists, or implementation checklists (those belong in design docs or PR description).

## When to add a change record

One record per coherent batch of REQ changes (feature slice). Skip typo-only edits that do not change behavior.

## Cross-domain changes

Use **one change file** that lists all touched REQs across domains. Do not split into per-domain changelog files.

## Validation

From the repository root:

```bash
npm run req:validate
```

Checks: atomic REQ frontmatter, broken `REQ-*` references, shared docs present, deprecated monolith files absent, change records reference valid IDs. Regenerates `docs/req/README.md` and `.req-manifest.json`.

## For AI agents

1. If `docs/req/changes/` has files with **Status:** `pending`, read them plus referenced REQ and `_shared` paths.
2. Otherwise use `docs/req/README.md` to locate relevant atomic REQ files.
3. Cross-cutting terms and gate ordering: read `docs/req/_shared/` before inferring behavior from a single REQ.
